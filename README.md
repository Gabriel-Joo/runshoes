# RUNSHOES

> 러닝화 스펙을 다 보여주되, 모르는 말이 없게.
> 초보 러너를 위한 러닝화 스펙 & 리뷰 아카이브

React · TypeScript · json-server 기반 번외 산출물 PART_02

---

## 1. 기획 배경

기존 러닝화 정보는 대부분 상급자 기준으로 쓰여 있다.
드롭, 스택하이트, 프로네이션 같은 용어가 설명 없이 나열되어 있어
초보 러너는 정작 "나에게 맞는 신발인지"를 판단할 수 없다.

RUNSHOES는 **정보를 덜어내지 않는다.**
대신 모든 전문 용어에 즉시 확인 가능한 설명(툴팁)을 붙여
초보와 상급자가 같은 페이지를 볼 수 있게 한다.

### 차별점

| | 내용 |
|---|---|
| 용어 툴팁 | 스펙 옆 `ⓘ` 클릭 시 용어 설명 표시 (`terms` 데이터 기반) |
| 수치 + 해석 병기 | `210g · 가벼운 편`, `4mm · 낮음` |
| 3-Step 맞춤 추천 | 페이스 / 발볼 / 카본 유무 3문항 → 조건에 맞는 러닝화 필터링 |

### 확장 로드맵 (PoV 이후)

```
        [ 러닝 통합 플랫폼 ]
              │
   ┌──────────┼──────────┐
  신발        코스        대회
 (PoV)      (예정)      (예정)
   │
 리뷰·평점·찜  ← 엔진 공유
```

세 도메인 모두 "항목 + 리뷰 + 평점 + 찜" 구조가 동일하다.
이를 위해 아래 세 가지를 미리 설계에 반영한다.

1. `reviews`에 `targetType` 필드 — 나중에 `course`, `race` 추가 가능
2. 컴포넌트를 도메인에 묶지 않음 — `ShoeCard`가 아닌 `ItemCard`
3. 폴더를 도메인 기준으로 분리 가능한 구조 유지

## 2. 요건 체크리스트

> 출처: 번외 산출물 안내 PART_02 (12~18p)

### ① 목록 화면 `/` — 단일 화면 구조 유지

- [ ] 항목 리스트 렌더링 — `GET`
- [ ] 카테고리 필터 (버튼 탭) — **용도별 5개**
- [ ] 정렬 (평점순 · 리뷰많은순 · 좋아요순) — **`?_sort=&_order=` 서버 정렬**
- [ ] 좋아요·찜 토글 → 서버 저장, 새로고침해도 유지 — `PATCH`
- [ ] 찜한 항목만 보기 — **온/오프 토글, 별도 페이지 아님**
- [ ] 항목 클릭 → 상세 모달

### ② 등록·관리 `/new`, `/edit/:id`

- [ ] 새 항목 등록 — `POST`
- [ ] 기존 항목 수정 (`useParams`로 id) — `PATCH`
- [ ] 항목 삭제 — `DELETE`
- [ ] 저장·삭제 후 목록으로 이동 — `useNavigate`

### ③ 베스트 `/best`

- [ ] 평점 · 리뷰 · 좋아요 기준 정렬 (`GET` + 정렬)
- [ ] 1~5위 랭킹 카드 UI

### ④ 공통 · 라우팅

- [ ] 상단 메뉴(`Link`)로 화면 이동
- [ ] `Routes` · `Route`로 분기
- [ ] (선택) NotFound `path="*"`

### ⑤ 상세 모달 — 목록 화면에서 사용

- [ ] 주제별 상세 정보 표시 — **스펙 + 용어 툴팁**
- [ ] 리뷰 목록 (작성자 · 별점 · 내용) — `GET`
- [ ] 평점·리뷰 작성 폼 → 등록 후 다시 불러오기 — `POST` + re-fetch
- [ ] 리뷰 삭제 — `DELETE`
- [ ] 모달 열기 / 닫기 — **배경 클릭 닫기 포함**
- [ ] (선택) 등록·관리 화면의 삭제 확인 모달

### ⑥ 서버 연결 (json-server)

- [ ] json-server 로컬 설치 후 `yarn`/`npx`로 실행
- [ ] Vite와 터미널 **2개 동시** 실행
- [ ] 데이터 = 항목(`shoes`) + 리뷰(`reviews`, 항목별 구분)
- [ ] 쓰기 후 다시 불러오기 (re-fetch)
- [ ] TS `interface`로 모양 정의, `useState<T[]>`

### ⑦ 반응형

- [ ] 웹 / 모바일 화면 크기에 따라 자연스럽게 전환
- [ ] 레이아웃 형태 자유 (그리드 · 리스트 · 테이블 무관)
- [ ] 목록 → 상세 모달 흐름은 **모든 화면 크기에서 유지**
- [ ] 모달이 모바일에서 깨지지 않도록 폭 · 스크롤 처리

### 놓치기 쉬운 항목

1. 찜 목록은 **별도 페이지가 아니라 토글** (추가 페이지는 확장으로만)
2. 정렬은 **클라이언트 `.sort()`가 아니라 서버 쿼리스트링**
3. 리뷰 등록 후 **로컬 상태 push가 아니라 re-fetch**
4. 모달 **배경 클릭 닫기** 명시됨

## 3. 데이터 설계

### shoes

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | number | |
| `brand` | string | 나이키 · 아식스 · 호카 · 아디다스 · 뉴발란스 · 온 · 브룩스 |
| `model` | string | 모델명 |
| `purpose` | string | **필터축** — 조깅·회복주 / 템포·인터벌 / LSD·장거리 / 레이스 / 트레일 |
| `stability` | string | 중립화 / 안정화 |
| `weight` | number | 무게 (g) |
| `drop` | number | 드롭 (mm) |
| `width` | string | 발볼 — 좁음 / 보통 / 넓음 |
| `carbon` | boolean | 카본 플레이트 유무 |
| `stackHeight` | number | 미드솔 두께 (mm) — 쿠션의 양 |
| `price` | number | 정가 (원) |
| `image` | string | `/images/shoe1.jpg` |
| `description` | string | 소개 문구 |
| `rating` | number | 평균 별점 — **정렬용으로 저장** |
| `reviewCount` | number | 리뷰 수 — **정렬용으로 저장** |
| `liked` | boolean | 찜 여부 |

> `rating` · `reviewCount`를 저장하는 이유:
> 요건이 `?_sort=&_order=` **서버 정렬**을 명시하므로
> json-server가 인식할 수 있는 실제 필드여야 한다.
> 리뷰 `POST` / `DELETE` 시 해당 신발도 `PATCH`로 갱신한다.

## 4. 용어 정리 (작성 중)

**드롭 (Drop)**
뒤꿈치와 앞꿈치의 높이 차이(mm). 클수록 뒤꿈치 착지가 편하고 무릎 부담이 줄지만,
낮으면 종아리·아킬레스건을 더 쓰게 되어 적응 기간이 필요하다.
시중 제품은 대체로 6~12mm이며 초보는 8~10mm가 무난하다.

**스택하이트 (Stack Height)**
미드솔의 두께(mm). 발밑에 깔린 쿠션의 양을 뜻한다.
30mm 이하면 얇은 편, 40mm 이상이면 두꺼운 편이다.
두꺼울수록 푹신하고 장거리에 유리하지만, 무게가 늘고 지면 감각이 둔해진다.

**프로네이션 (Pronation)**
착지 시 발이 안쪽으로 살짝 기울어지는 자연스러운 움직임. 충격을 흡수하는 정상 동작이다.
이 움직임이 과도한 경우를 **과내전(오버프로네이션)** 이라 한다.

**중립화 (Neutral)**
교정 장치 없이 쿠션을 균등하게 배치한 신발. 시중 러닝화 대부분이 여기 해당한다.
흔히 말하는 "쿠션화"와 같은 말이며, 쿠션이 더 많다는 뜻은 아니다.

**안정화 (Stability)**
미드솔 안쪽에 단단한 소재를 넣어 발이 안쪽으로 무너지는 것을 막아주는 신발.
과내전인 사람에게 필요하며, **해당되지 않는 사람이 신으면 오히려 불편하다.**
안정화가 더 좋은 신발이 아니라 발 유형에 맞춰 고르는 것이다.

**트레일 러닝 (Trail Running)**
포장도로가 아닌 산길·흙길·자갈길을 달리는 러닝.
트레일화는 밑창 돌기가 깊고, 어퍼가 튼튼하며, 앞코에 토캡 보강이 들어가 무게가 무거운 편이다.

**카본 플레이트**
미드솔에 삽입된 탄성판. 반발력을 크게 높여주지만 가격이 비싸고
근력이 충분하지 않은 초보에게는 부담이 될 수 있다.

## 5. 개발 환경

```
~/workspaces/frontend/react/runshoes/
```

### 설치

```bash
yarn create vite runshoes --template react-ts
cd runshoes
yarn
yarn add react-router-dom
yarn add -D json-server@0.17.4
```

> json-server는 **0.17.4 고정.**
> 1.x는 id를 nanoid 문자열로 생성해 `/edit/:id` 라우팅과 맞지 않는다.

### 실행 — 터미널 2개

```bash
yarn dev       # Vite      http://localhost:5173
yarn server    # json-server  http://localhost:3000
```

### 폴더 구조

```
runshoes/
├─ db.json              ← 루트에 위치 (src/ 아님)
├─ package.json
├─ public/
│  └─ images/           ← 신발 이미지
└─ src/
   ├─ types/            ← Shoe, Review, Term interface
   ├─ components/       ← ItemCard, Modal, TermTooltip
   ├─ pages/            ← ShoeList, Best, ShoeForm, NotFound
   ├─ App.tsx
   └─ main.tsx
```
